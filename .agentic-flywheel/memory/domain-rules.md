# Domain Rules

Recurring rules for common work types.

## Public Incident Writeups

- Prefer the phrasing "`<user-facing problem>` caused by or related to `<suspected mechanism>`" over naming the incident solely after an internal component.
- Avoid single-cause claims unless the source is strong enough to support them.
- If multiple caching layers may explain delayed recovery, name them separately instead of collapsing them into one bucket.
- When citing operational behavior from a vendor, link the relevant source close to the claim.

## Iterative Writing Tasks

- Compare the first draft with the accepted draft and ask what changed in the harness, not only in the output.
- Promote repeated wording preferences into memory when they generalize across future work.
